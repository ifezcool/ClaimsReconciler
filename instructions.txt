# Claims Reconciliation App — Review & Fix Instructions

Context for the agent: this is a Streamlit app (`streamlit_app.py` as entry point) that reconciles claims data between a Claims department and a Finance department, compiles per-category submission files (Telemedicine, Ambulance, Appeals), and uploads cleaned data into SQL Server tables via pyodbc. Work through the items below in order — Critical first.

## CRITICAL — bugs and security

1. **Email CC list bug in `utils.py`** (`send_variance_email`): the list literal is missing a comma —
   `cc_email = ["ifeoluwa.adeniyi@avonhealthcare.com"  "adedamola.ayeni@avonhealthcare.com", ...]`
   Python silently concatenates the first two strings into one invalid address. Fix the missing comma and add a unit test or assertion that validates every entry in `cc_email`/`cc_list` looks like a valid email before sending.

3. **Row-insert loop aborts on first failure instead of skipping it.** In `AmbulanceUpload.py`, `AppealsUpload.py`, `TelemedicineUpload.py`, and `DB_Upload.py`, the per-row insert loop does:
   ```python
   except Exception as e:
       st.error(f"Row {i+1} failed: {e}")
       break
   ```
   `break` stops processing every subsequent row in the file, so one bad row silently discards the rest of the upload while reporting only partial success. Change to `continue`, accumulate a list of failed rows with their error messages, show that list at the end, and don't `commit()` until you've decided how to handle partial failures (see #4).

4. **No transaction safety around bulk inserts.** All four upload scripts call `conn.commit()` once at the end regardless of how many rows failed. Wrap the insert batch in an explicit transaction; on a fatal error, roll back rather than partially committing. Consider switching to `cursor.executemany()` / `fast_executemany=True` (pyodbc) for both speed and to make the row-level error story explicit rather than emergent from a Python for-loop.

5. **Hardcoded recipient lists scattered across 5 files** (`ambulance.py`, `appeals_page.py`, `telemedicine.py`, `session_manager.py`, `utils.py`), each slightly different. Move all recipient/CC lists into `secrets.env` (or a config module) as a single source of truth, e.g. `NOTIFY_TO`, `NOTIFY_CC_CLAIMS`, `NOTIFY_CC_FINANCE`. This avoids the current risk of one list getting updated and the others drifting.

## HIGH — eliminate the duplicated modules

This is the most impactful change. Right now:
- `AmbulanceUpload.py`, `AppealsUpload.py`, `TelemedicineUpload.py` (and most of `DB_Upload.py`) are the same script three/four times over, differing only in table name, a few labels, and the `column_mapping` dict.
- `ambulance.py`, `appeals_page.py`, `telemedicine.py` are the same compilation/comparison/email-notification logic three times over.
- `instructions.txt` literally documents a manual find-and-replace process for creating the next copy. That's a maintenance liability: every future bug fix (like #3 above) has to be applied N times by hand and will eventually drift.

Refactor instructions:
- Create `db_upload_common.py` exporting one function, e.g. `render_generic_upload(table_name, column_mapping, date_columns, numeric_columns, consolidate_target)`. Each of `AmbulanceUpload.py` etc. becomes a 3-line wrapper that calls it with that category's config.
- Create `compilation_common.py` exporting `compile_files(...)`, `create_compiled_excel(...)`, `compare_with_finance(...)`, `send_notification_email(...)`, `extract_schedule_from_filename(...)`, `get_download_link(...)`, and `show_compilation_page(label, session_keys, ...)`, all parameterized by category name/label instead of hardcoded "telemedicine"/"ambulance"/"appeals" strings. `telemedicine.py`, `ambulance.py`, `appeals_page.py` become thin wrappers.
- Once this exists, adding a new claim category (e.g. "Dental") should mean adding one config entry, not duplicating ~300 lines of code.

## MEDIUM — data integrity & performance

7. **All non-date DB columns are `VARCHAR(MAX)`**, including amount and count fields (`AMOUNT_RECOMMENDED_FOR_PAYMENT_N`, `APPROVED_PA_VALUE_N`, `PA_AMOUNT`, `AMOUNT_CLAIMED`, `NO_OF_UNITS`, etc.). This makes SQL-side aggregation, sorting, and BI reporting on these tables harder/slower and loses type safety. Use `DECIMAL(18,2)` for money fields and `INT` for counts; only keep free-text fields as `VARCHAR`.

8. **The "consolidate" step is a manual SSMS copy-paste** (`st.code("INSERT INTO Compiled_X SELECT * FROM xtbl")` shown to the user after every upload). Automate this: after a successful upload (and ideally behind a confirmation checkbox), run the same `INSERT INTO ... SELECT * FROM ...` from the app itself.

9. **No automatic truncate before upload** — the file uploader labels just say "please truncate before uploading," relying entirely on the user remembering. Add a checkbox ("I have truncated this table" / "Truncate now") that the app can act on directly, or have the app check `SELECT COUNT(*) FROM table` and warn if it's non-empty before inserting.

10. **No schema validation before insert.** If an expected Excel column is missing, the code just inserts `None` for that field silently (only `DB_Upload.py` surfaces a warning via `missing_in_mapping`/`missing_in_excel`). Add the same up-front "expected vs. found columns" check to the other three upload scripts, and refuse to proceed (or require explicit override) if required columns are absent.

## LOW — cleanup

11. Replace the manual base64 `<a href="data:...">` download links (`get_download_link` in all three compilation files) with `st.download_button`, which is already used correctly elsewhere in `streamlit_app.py` and avoids `unsafe_allow_html=True`.
12. Add basic logging (Python `logging` to file or a DB audit table) for who uploaded what and when — useful for a financial reconciliation tool and currently absent; only `print()` debug statements exist.
13. Pin `requirements.txt` to exact versions you've tested (`==`) rather than `>=`, to avoid a future `pip install` silently picking up a breaking pandas/streamlit release.
14. Add a handful of unit tests for the pure-logic helpers in `utils.py` (`extract_schedule_data`, `find_missing_schedules`, `calculate_schedule_amounts`, `generate_reconciliation_report`) and for the `clean_value`/date-parsing functions, since these drive financial figures and currently have zero test coverage.
